Here begins a series of blog posts combining two concepts I've long since been
interested in: **procedural generation** and **functional programming**.

[Procedural generation
(procgen)](https://en.wikipedia.org/wiki/Procedural_generation) is the technique
of creating data algorithmically instead of manually. It's typically powered by
randomness operating within certain constraints. This allows you to generate
many different instances of a structure which all follow a coherent pattern.

I'm utterly fascinated by the idea of giving a computer a set of rules and a
random number generator and it giving you something beautiful in return! For
example, the village map below was [generated
procedurally](https://watabou.itch.io/village-generator). Favourite haunts of
mine include [r/proceduralgeneration on
Reddit](https://www.reddit.com/r/proceduralgeneration/) and [watabou on
itch.io](https://watabou.itch.io/).

![Procedurally generated village](/assets/images/2024-12-05-PGF-0/village.png)

[Functional programming
(FP)](https://en.wikipedia.org/wiki/Functional_programming) is a programming
paradigm which centres on the idea of composing functions. A key concept within
FP is that, in general, functions should be _pure_: they must be _deterministic_
(a given input always produces the same output) and they must not cause
_side-effects_ (observable behaviour apart from the return value of the
function). FP values _immutability_ (the idea that state should not change value
once they have been initialised) very highly. TODO fewer brackets

At first glance it might seem odd to combine procgen and FP, because randomness
is an inherently impure concept (by definition, you don't always get the same
answer when you ask for a random number). However, by embracing FP concepts in
the core of your code and pushing all the impurity to the edges (what Mark
Seemann calls an [impureim
sandwich](https://blog.ploeh.dk/2020/03/02/impureim-sandwich/)) you can gain
many of the benefits of FP whilst still being able to generate random data. At
least, that's the theory; read on to find out if it holds true!

As a .NET developer, you might think that I'd choose F#, a functional-first
language, for a series involving FP. However, I'm going to use C# in this
series, for the following reasons:

1. C# is by far the language I am most familiar with.
2. tooling is better
3. c# has lots of functional stuff in it

Perhaps a post later on in this series could explore these ideas using F#.
